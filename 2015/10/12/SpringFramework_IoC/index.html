Inversion of Control(IoC)

發表於 2015-10-12   |   分類於 Web ， Java ， framwork ， spring ， 名詞定義

Overview

spring 核心功能, 中文翻譯為控制反轉，

是spring強調的重要功能之一。

example:

想像我們有一個 HelloWorld的類別物件，有一個hello的方法，呼叫該方法後會印出 hello!的文字。

HelloWorld.class

public class HelloWorld {

    public void hello(){
        System.print.out("hello!");
    }

}

一般來說我們要使用該物件時候以下面的方式來寫

public class BOImpl {

    public void sayHello(){
        HelloWorld theHelloWorld = new HelloWorld();
        theHelloWorld.hello();
    }
}

而spring 的 控制反轉 就是不再讓開發人員自行new HelloWorld();

由spring container 透過 我們的.xml設定檔宣告bean,

以Dependency Injection (依賴注入)到相關的程式內，

我們就可以直接的使用該物件的方法，而不必去new實體。

因為spring幫我們把這段作業處理完畢。

public interface BO {
    public void sayHello(HelloWorld iHelloWorld);
}


public class BOImpl {
    HelloWorld theHelloWorld;

    public void sayHello(HelloWorld iHelloWorld){
        this.theHelloWorld = iHelloWorld;
        theHelloWorld.hello();
    }
}

IoC 要求的是container不侵入應用程式本身，應用程式本身提供Interface，如上方的

public void sayHello(HelloWorld iHelloWorld);

container可以透過這些interface將所需的資源Dependency Injection至程式中，

應用程式不向container主動要求資源，故而不會依賴於container的元件，

應用程式本身不會意識到正被container使用，

public void sayHello(HelloWorld iHelloWorld){
    //對這段程式碼來說只要用 iHelloWorld就可以呼叫hello()
    //不會去管iHelloWorld怎麼來    
    this.theHelloWorld = iHelloWorld;
    theHelloWorld.hello();
}

可以隨時從container中脫離轉移而不用作任何的修改，

而這個特性正是一些業務邏輯(Business Logic)最需要的。

所謂的IoC,我的理解是:

將你設計好的class交給spring去控制，而不是在你的calss內自己控制

而背後這樣設計的用意是讓我們所寫的業務邏輯(Business Logic)程式碼可以重複使用。
