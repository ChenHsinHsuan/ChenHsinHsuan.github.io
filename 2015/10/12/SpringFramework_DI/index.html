Dependency Injection(DI)

發表於 2015-10-12   |   分類於 Web ， Java ， framwork ， spring ， 名詞定義

Overview

Spring 所採用的是Dependency Injection 來實現 IoC

保留抽象介面，讓組件依賴於抽象介面，當組件要與其它實際的物件發生依賴關係時，透過抽象介面來注入依賴的實際物件。

example:

void saveNotice() { 

    saveAInquery() 

}

對於上面這段程式碼:saveNotice()就是所謂的高層模組,相對的saveInquery()就是所謂的低層模組

saveNotice()這個方法必須依賴於 saveInquery(),

如果今天我們想要存的是一張訂單,那麼這支程式就沒辦法重複使用了,

從上面這個例子我們發現

低層模組saveAInquery()的變動, 使高層模組saveNotice()也必須跟著更動，這不是一個好的設計方式

在設計上希望模組都依賴於模組的抽象，這樣才可以重用高層的業務設計

若改寫成：

public class BO { 

    private InqueryWriter writer = new InqueryWriter(); 

    public void saveNotice() { 

        writer.save(); 

    } 
}

BusinessObject的存檔對象決定權在於writer是誰,

但是這樣寫程式仍然沒辦法重複使用,

如果透過interface的宣告就可以有效改變這樣的局面

public interface NoticeWriter { 
    public void save(); 
} 



public class BO { 
    private NoticeWriter writer; 

    public void setDeviceWriter(NoticeWriter writer) { 
        this.writer = writer; 
    } 

    public void saveNotice() { 
        writer.save(); 
    } 
}

這樣一來，BusinessObject 就是可重用的，如果今天有存儲至 Inquery 或 Order 的需求時，只要實作 NoticeWriter 即可，而不用修改 BusinessObject

而真正的實作如下

public class InqueryWriter implement NoticeWriter { 
    public void save() { 
        .... 
        // 實際儲存至Inquery的程式碼 
    } 
} 

public class OrderWriter implement NoticeWriter { 
    public void save() { 
        .... 
        // 實際儲存至Order的程式碼 
    } 
}

injection又有3種區別:
Martin Fowler的文章中談到了三種實現方式

  1. Interface injection (Type 1 IoC)
  2. Setter injection (Type 2 IoC)
  3. Constructor injection (Type 3 IoC)

Interface injection (Type 1 IoC)

使用Type 1 IoC時會要求實作介面，這個介面是為容器所用的，

容器知道介面上所規定的方法，它可以呼叫實作介面的物件來完成依賴關係的注入，例如:

public interface BO { 
    public void createDependency(Map dependObjects); 
} 

public class BOImpl implement BO { 
    private BidBO bidBO; 

    public void createDependency(Map dependObjects){
        this.bidBO = dependObjects; 
        // 在這邊實現與BOImpl的依賴關係 
        ...... 
    } 

    public void sendInquery() {  
        ...... 
        bidBO.saveAInquery(); 
    } 
}

Type 1 IoC要求BO實現特定的介面，這就使得BusinessObject依賴於容器，

如果日後BO要脫離 Spring Container，就必須修改程式，

想想在更複雜的依賴關係中產生更多複雜的介面，

組件與容器（框架）的依賴會更加複雜，最後使得組件無法從容器中脫離。

所以Type 1 IoC具有強的侵入性，使用它來實現依賴注入會使得組件相依於容器（框架），降低組件的重用性。

Setter injection (Type 2 IoC)

public class BO { 

    private NoticeWriter writer; 

    public void setDeviceWriter(NoticeWriter writer) { 
        this.writer = writer; 
    } 

    public void saveNotice() { 
        writer.save(); 
    } 
}

上方這樣的寫法就是 Type 2 的 IoC, Setter injection

也是spring最鼓勵的injection的方式

Constructor injection (Type 3 IoC)

public class BO { 
    private NoticeWriter writer; 

    public BusinessObject(NoticeWriter writer) { 
        this.writer = writer; 
    } 

    public void save() { 
        writer.save(); 
    } 
}

使用 Constructor 的好處是：可以在new物件的同時一併完成依賴關係的建立，

但是如果要建立的物件關係很多，則會在Constructor上留下一長串的參數，

這時使用 Setter 會是個不錯的選擇，

另一方面，Setter 可以有明確的名稱可以瞭解注入的物件會是什麼，

像是setXXX()這樣的名稱會比Constructor上用參數位置代表某個物件來得好。

官方的說法是

setter的注入缺點是無法明確表示哪些屬性是必須注入的,哪些是選擇性的注入

constructor的注入方式反而可以有效解決這樣的問題，如果參數於xml檔填寫的不完整，就無法使用該bean

Dependency Injection 結論就是

程式不依賴於實作，而是程式與實作都要依賴於抽象
